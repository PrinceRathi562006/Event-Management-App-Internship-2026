Event Organizer - Project Documentation

1. Overview

Event Organizer is a MERN stack campus event management system for Students,
Organizers, and Admins. It supports OTP-based registration, date of birth in
profiles, organizer approval, event approval, public event discovery, paid/free
bookings, Razorpay payments, QR tickets, QR attendance, coordinator requests,
feedback, expandable notifications, branded certificates, profile/settings,
chart-based analytics dashboards, AI-assisted event creation, poster export,
seat selection, waitlists, rich event pages, social sharing, Google Calendar
links, PWA installation, dark/light theme switching, volunteer operations,
scan history, exports, reports, MongoDB-backed realtime organizer chat,
editable volunteer settings, second-level countdowns, and gamified student profiles.
Public event, featured event, calendar, poster, speaker, discussion, volunteer,
and scan-history data are loaded from backend/user-created records only; the
frontend does not ship sample event fallbacks.

3. User Roles

Student:
- Registers with full academic details, date of birth, and profile image.
- Account is created only after email OTP verification.
- Books approved events.
- Receives QR ticket by email after event registration.
- Pays exact event fee for paid events through Razorpay.
- Gets attendance marked by QR scan.
- Receives certificate only if attendance is Present.
- Can accept or reject student coordinator requests.
- Accepted student coordinators can open event settings, view attendance rosters, mark QR attendance, and generate certificates for their assigned coordinator events.

Organizer:
- Registers with department, designation, date of birth, and profile image.
- Account is created only after OTP verification.
- Requires admin approval before organizer tools are enabled.
- Creates events, adds speakers, uploads banners/gallery/signature, adds partner companies, sets exact fee, and sets last date of registration.
- Uses local AI assistance to generate description, agenda, learning outcomes, and requirements.
- Generates a downloadable event poster from title, date, venue, theme, and category.
- Adds timeline entries, badge types, and richer event content.
- Sends coordinator requests to students and approved organizers.
- Manages own/assigned/coordinated events.
- Scans QR tickets and generates certificates.
- Uses editable volunteer assignment, QR scanner history, CSV export, completion report panels, and navbar Organizer Chat.

Admin:
- Full platform access.
- Approves organizers.
- Approves or rejects events.
- Creates events.
- Manages users and analytics.
- Views attendance heatmap, admin analytics, operations panels, exports, cancellation/refund status, and navbar Organizer Chat.

4. Registration Flow

1. User fills registration form.
2. Server checks duplicate email and phone.
3. Registration data is stored temporarily with OTP.
4. No User document is created before OTP verification.
5. User verifies OTP.
6. User document is created in MongoDB.
7. Organizer accounts start as Pending Approval.

5. Event Flow

1. Approved organizer or admin opens Create Event.
2. Event details, banner, gallery, speakers, capacity, last date of registration, fee, certificate signature, and partner companies are saved.
3. Organizer can generate AI event copy and poster assets before submission.
4. Organizer can add agenda, learning outcomes, requirements, timeline, badges, speakers, sponsors/partners, and gallery assets.
5. Organizer-created events are Pending.
6. Admin reviews and approves/rejects.
7. Only Approved and Published events appear publicly.
8. Event Settings can update event details, last date of registration, certificate signature, and partner companies later.

6. Coordinator Flow

1. Organizer/admin searches students by name, email, or roll number.
2. Organizer/admin searches organizers by name, email, or department.
3. Selected users receive coordinator requests.
4. Requests appear in the Coordinator Requests panel on the student or organizer dashboard.
5. Student and organizer users accept or reject pending requests from that panel.
6. Accepted student coordinators are saved as Student Coordinators and receive coordinator event access.
7. Accepted organizer coordinators are saved as Organizer Coordinators and Assigned Organizers.
8. Coordinators can open assigned event details, use the Manage Event button, edit event settings, view registrations, view analytics, use volunteer/scanner-history panels, export CSV, print reports, mark attendance, manage feedback replies/visibility, use navbar Organizer Chat, and generate certificates for assigned events.

7A. Organizer Chat

1. Organizer Chat appears in the navbar after login only for admins, approved organizers, and users assigned as coordinators to at least one event.
2. Chat messages are saved in MongoDB through OrganizerChatMessage.
3. Socket.io broadcasts new messages live to the organizer-chat room.
4. Users can send text, reply to previous messages, and upload images/documents/files as attachments.
5. Attachments are stored under backend uploads/chat and served through the existing uploads static route.
6. Dashboard no longer contains Organizer Chat; dashboard keeps Volunteer Management as editable event operations.
7. The frontend loads chat history from /api/chat/organizer/messages and then listens for live organizer-chat-message events.
8. The frontend dynamically loads the Socket.io client from the backend so no separate client package is required.
9. Socket room access is protected on the server, so only eligible users can join organizer-chat.

7. Booking, QR, And Payment

1. Student books an approved event.
2. System creates a Booking ID like EVT-2026-000123.
3. QR code is generated with Booking ID, User ID, Event ID, and Ticket Number.
4. Student receives a registration email with event details and attached QR ticket image.
5. Free events are marked Paid automatically.
6. Paid events use the exact fee added by organizer/admin.
7. Razorpay order amount is created from the saved booking amount.
8. Students can choose a preferred seat number during registration.
9. If seats are full, students can join the waitlist instead of registering.
10. When a booking is cancelled, the oldest waitlisted booking is promoted and receives a Seat Available notification.
11. Paid booking cancellation stores Refund Initiated status.
12. Event cancellation marks paid confirmed bookings as Cancelled and Refund Initiated.

8. Attendance And Certificates

1. Assigned organizers, student coordinators, organizer coordinators, and admins scan QR tickets.
2. Booking is verified.
3. Duplicate scans are blocked.
4. Attendance is marked Present.
5. Organizer/admin/coordinator attendance roster shows Total Registered, Present Students, and Absent Students.
6. Clicking a student opens full student and booking details.
7. Organizer/admin/coordinator generates certificate after attendance is Present.
8. Certificate PDF includes organizer signature image and partner companies when they are added to the event.
9. Certificate PDF is saved under backend uploads.
10. Student receives an email with download link and attached PDF.
11. Certificate verification is supported through QR/booking identity on generated certificate and ticket flows.
12. Printable ID-card style data is available through student/booking details and QR ticket information.

8A. Notifications

1. Notifications load in the dashboard and navbar.
2. Dashboard notification rows can be clicked to open full notification details.
3. Notification detail includes message, event title, event date, venue, status, and action links when available.
4. Opening a notification marks it as read.

- Dashboard: role-based stats and actions, event detail links, chart-based analytics, registration trend analysis, admin analytics, attendance heatmap, user-added volunteer management, backend scan history, exports, total registered/present/absent attendance summaries, and present/upcoming/past event labels.
- Dashboard: coordinator request accept/reject panel for students and organizers.
- Dashboard: profile details card with a View Profile button.
- Create Event: dedicated event creation page with AI description generator, poster generator, timeline, badges, last date of registration, certificate signature, and partner company fields.
- Event Settings: update event, last date of registration, certificate branding, and send coordinator requests.
- Profile: read-only account details including date of birth, profile completion progress, resume integration guidance, and achievements.
- Settings: editable profile/account details including date of birth.
- Admin: users, approvals, analytics, attendance roster, event management.
- Global UI: dark/light theme switching with mode-aware text/background colors, navbar Organizer Chat, and PWA install support.

10. Deployment Checklist

Backend environment variables:
- PORT
- NODE_ENV
- MONGO_URI
- JWT_SECRET
- JWT_EXPIRE
- CLIENT_URL
- API_RATE_LIMIT_MAX
- SUPPORT_EMAIL
- EMAIL_USER
- EMAIL_PASS
- RAZORPAY_KEY_ID
- RAZORPAY_KEY_SECRET
- CLOUDINARY_CLOUD_NAME
- CLOUDINARY_API_KEY
- CLOUDINARY_API_SECRET

Frontend environment variables:
- VITE_API_URL
- VITE_RAZORPAY_KEY_ID

Before deployment:
- Use a production MongoDB connection string.
- Use a long random JWT secret.
- Set CLIENT_URL to deployed frontend URL. Multiple frontend URLs can be comma-separated.
- Set VITE_API_URL to deployed backend API URL when frontend and backend are on different domains.
- If VITE_API_URL is not set in production, the frontend uses same-origin /api instead of localhost:5000.
- Add Razorpay live keys for production.
- Add Cloudinary keys for uploads.
- Add Gmail app password or SMTP credentials for emails.
- Rotate any keys that were committed or shared during development.

12. Known Local Development Notes

- In local development, start the backend or set VITE_API_URL before using the frontend.
- In production, configure VITE_API_URL for a separate backend domain or deploy the frontend behind the same domain so /api reaches the backend.
- Vite/React development warnings about preloaded resources or WebSocket back-forward cache are browser/dev-server warnings, not production API failures.
- If emails are not configured, development mode logs email content instead of sending.
